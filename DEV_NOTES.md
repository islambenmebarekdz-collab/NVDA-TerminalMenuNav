# TerminalMenuNav — ملاحظات التطوير

## المشكلة والأساس التجريبي
قوائم CLI التفاعلية (inquirer وأمثالها) تعيد رسم كتلة الخيارات كاملة بكل ضغطة
سهم عبر ANSI. تشخيص 2026-07-11 بسجل NVDA (مستوى input/output) على
`C:\Users\benme\TerminalTest\menu_test.py` أثبت:

- **نمط A (إعادة رسم الكتلة كلها — السائد):** كل سهم ينطق (1) سطر مؤشر
  الطرفية وهو ثابت ومضلل (آخر خيار دائماً، لأن المؤشر مركون تحت الكتلة)،
  ثم (2) الكتلة كاملة. لا يمكن معرفة الموضع.
- **نمط B (سطران فقط):** أخف؛ يُنطق السطر القديم ثم الجديد، مع نفس السطر المضلل.
- **نمط C (المؤشر على سطر الخيار):** أول نطق بعد كل سهم هو الخيار الصحيح —
  أي أن تتبع NVDA لسطر المؤشر موثوق. الإضافة تجعل كل قائمة تتصرف كأنها نمط C.

## المعمارية: "اقرأ الحالة لا الحدث"
- globalPlugin يسجّل في `speech.extensions.filter_speechSequence`
  (متوفر منذ NVDA 2023.2؛ يوجد حارس رسمي `if not speechSequence: return`
  بعد الفلتر مباشرة في speech.py، فإرجاع `[]` لكتم الكلام آمن).
- الإعلان مجرد **مشغّل**. عند إعلان "ذي صلة" تُقرأ حالة المخزن الفعلية
  (`makeTextInfo(POSITION_ALL)`، آخر 200 سطر) ويُبحث فيها عن القائمة —
  فلا يهم نمط إعادة الرسم إطلاقاً.
- **بوابة الصلة** (رخيصة، قبل أي قراءة): علامة مؤشر في أي موضع من النص
  (الإعلانات تصل مسطّحة من VS Code — لا line-anchoring!) أو تطابق سطرٍ
  حرفياً مع عنصر من آخر قائمة مكتشفة (يلتقط إعلان سطر المؤشر المضلل الذي
  لا يحمل علامة).
- **كشف القائمة** (menuDetect.py، نقي وقابل للاختبار): آخر سطر مؤشر في
  المخزن ← توسيع كتلة متصلة من أسطر "شكلها خيار" ← شروط: ≥2 خيار، مؤشر
  واحد بالضبط، نهاية الكتلة ضمن آخر سطرٍ غير فارغ أو سطرٍ واحد بعدها
  (MAX_LINES_FROM_BOTTOM=1؛ يسمح بسطر تلميح مثل "(Use arrow keys)"
  ويستبعد الإطارات المنتهية التي دُفعت للأعلى بأسطر النتيجة).
- **منع التكرار:** مفتاح (selectedIndex, items)؛ الإعلان المزدوج لكل ضغطة
  (سطر المؤشر + diff إعادة الرسم) يُنطق مرة واحدة ويُكتم الثاني.
- **سياق القائمة الجديدة:** عند تغيّر items يُنطق السطر غير-الخيار الأقرب
  فوق الكتلة (السؤال) مرة واحدة قبل الخيار.
- **فشل آمن:** أي استثناء في الفلتر يعيد التسلسل الأصلي كما هو؛ وإن تعذّرت
  قراءة المخزن يُجرَّب تحليل نص الإعلان نفسه (fallback)، وإلا مرّ الكلام
  دون تغيير. الأسوأ الممكن = السلوك الحالي المزعج، لا أسوأ.

## تعريف الطرفية
role == TERMINAL (يغطي كونسول ويندوز الكلاسيكي وWindows Terminal مهما كان
appName للعملية داخله)، أو appName == "code" مع اسم يحتوي "erminal"
(طرفية VS Code كائن Chromium وليست TERMINAL). لا فلترة في محرر VS Code نفسه.

## الاختصارات والإعدادات
- NVDA+Alt+L تبديل، NVDA+Alt+K إعادة نطق الخيار الحالي.
  (محجوز في إضافات أخرى مثبتة — لا تستخدم: NVDA+Shift+M لـReviewMouseSync،
  NVDA+Alt+M لـVerbosityPlus، NVDA+Alt+N لـNavigationSounds، وكذلك
  NVDA+Alt مع z/x/i/t/v/c/h/PageUp/PageDown/applications.)
- لوحة إعدادات: enabled + announcePosition. config spec: terminalMenuNav.

## الترجمة
gettext قياسي: `addon/locale/ar/LC_MESSAGES/nvda.po` يُترجم بـ
`tools/build_mo.py` (msgfmt مصغّر بلا تبعيات، يستدعيه build.ps1). التحقق
محلياً: gettext.translation يحمّل الـ.mo والسلاسل مطابقة بايتياً.

- **اختبار انحداري حي:** فعّل تسجيل NVDA مؤقتاً:
  `& "C:\Program Files\NVDA\nvda.exe" -r --log-level=12`
  ثم شغّل `python C:\Users\benme\TerminalTest\menu_test.py` في طرفية VS Code

  لكل ضغطة ("GOLF-X ...، N من 5") بلا تكرار الكتلة وبلا سطر GOLF-5 المضلل.

## مخاطر معروفة / أعمال لاحقة
- قراءة POSITION_ALL في طرفية VS Code تحتاج تأكيداً حياً؛ إن أعادت نصاً
  فارغاً فسيعمل الـfallback فقط (الإعلانات المسطّحة لا تكفيه) والسلوك يبقى
  كالسابق — عندها الحل التالي هو قراءة شجرة xterm.js عبر كائن الوثيقة الأب.
- محث node/REPL "> " مسبوق بأسطر بادئة بمسافتين قد يعطي false positive.
- مراجعة يدوية لأسطر قائمة نشطة قد تُكتم كمكرر (مفتاح dedupe) — مقبول مبدئياً.
- قوائم "❯◉" الملتصقة (checkbox متعدد) غير مدعومة بعد.
- برايل غير مفلتر (الفلتر كلام فقط).

## معمارية v0.1.2: قراءة مؤجلة (debounce)
المشغّلات ذات الصلة تُبتلع فوراً وتعيد ضبط مؤقت 60ms؛ عند استقرار الدفعة
تُقرأ الحالة مرة واحدة ويُنطق الخيار الصحيح (ui.message محمية بعلم
_selfSpeaking من إعادة الفلترة). يحل السباق والتكرار معاً، ويجمع الضغطات
السريعة المتتالية في نطق أخير واحد. **شبكة أمان**: كل نص مبتلع يُحفظ في
_swallowed؛ إن لم تجد القراءة قائمةً أعيد نطقه حرفياً (بوابة خاطئة لا
تُكلف المستخدم إخراجاً حقيقياً أبداً).

## نتائج الاختبار الواقعي (2026-07-13، v0.1.3 مع questionary وClaude Code)
- **questionary select (مؤشر ❯): مثالي 100%** — أول اختبار على مكتبة حقيقية
  من عائلة inquirer، نجح دون أي تعديل، شامل الالتفاف والسياق والنتيجة.
- **questionary checkbox: عمل خلافاً للتوقع** — المؤشر "»" منفصل عن ○/●
  وضمن مجموعتنا. ثلاث شوائب أُصلحت في v0.1.4:
  1. سطر إخراج سابق فوق السؤال تسرّب كحدث caret شارد → توسيع البوابة إلى
     `above` (حتى 3 أسطر فوق الكتلة تُخزّن في Menu وتُكتم أصداؤها).
  2. رمز المؤشر وحده ("» ") نُطق عارياً → isBareMarker يبتلعه.
  3. ضغط المسافة (○→●) اعتُبر "قائمة جديدة" فأعاد نطق السؤال →
     isNewMenu يقارن العناصر بعد normalizeItem (تجريد رموز الاختيار)،
     ونطق حالة التبديل الجديدة يبقى ("● INDIA-4 nectarine، 4 من 4").
- **Claude Code CLI**: شهادة المستخدم "طبيعي وممتاز".
- درس جانبي: أوامر الاختبار للمستخدم يجب أن تكون سطراً واحداً قابلاً للصق
  مباشرة في PowerShell (لا `%TEMP%` ولا كلمات عربية داخل كتلة الكود).

## v0.2.0: دعم قوائم clack (نتيجة اختبار npm create vite، 2026-07-13)
اختبار npm create vite كشف **فشلاً كاملاً في الكشف** (عودة للفوضى القديمة):
@clack/prompts لا يرسم مؤشر سهم إطلاقاً — كل سطر يبدأ بحد صندوق "│"
والتحديد دائرة معبأة "●" مقابل "○" (◆/◇ أيقونات خطوات وليست خيارات).
الحل: **وضع كشف ثانٍ (radio-mode)** يُجرَّب بعد فشل وضع الأسهم:
- تجريد حدود الصناديق (_stripBox) قبل المطابقة.
- الكتلة = أسطر دوائر متجاورة، دائرة معبأة واحدة بالضبط (يرفض ملخصات
  checkbox متعددة ●)، سماحية أسفل أوسع (3) لأن الإطار يضم "..." وسطر
  تلميح "↑/↓ to navigate" وحد "└".
- أسطر "..." (نافذة متمررة) لا تدخل السياق ولا العناصر أبداً.
- isBareMarker وسّع لحدود الصناديق و"..." (تسريبات "└" و"»").
- الحقل الجديد below (حتى 3 أسطر تحت الكتلة) يُكتم صداه مثل above.
- isNewMenu: نفس السؤال ⇒ نفس القائمة (وإلا فمقارنة عناصر مجرّدة) — كي
  لا يُعاد نطق السؤال عند تمرير النافذة وتغيّر العناصر المرئية.
- الموضع المعلن في القوائم المتمررة هو ضمن النافذة المرئية فقط (قيد موثق).
درس منهجي: اختبار المحاكاة كشف علة السياق-"..." قبل وصولها للمستخدم.
